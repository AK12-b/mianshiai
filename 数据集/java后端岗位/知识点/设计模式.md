# 前⾔

软件设计原则有哪些？

在软件开发中，我们遵循⼀系列⾯向对象设计原则来提升代码的可维护性、可复⽤性和可扩展性，从⽽构建出更加健壮和灵活的系统。这些原则中最核⼼、最著名的就是 SOLID 原则，同时还有⼏个⾮常重要的补充原则。

# SOLID 原则：

# 1. 单⼀职责原则 (Single Responsibility Principle, SRP)

核⼼思想：⼀个类或模块应该有且只有⼀个引起它变化的原因。

通俗来说：⼀个类只做⼀件相关的事情。如果⼀个类承担了多个职责，那么当其中⼀个职责发⽣变化时，可能会影响到其他职责的实现，这会增加代码的复杂性和耦合度。

# 2. 开闭原则 (Open-Closed Principle, OCP)

核⼼思想：软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。

通俗来说：当需要增加新功能时，我们应该通过增加新的代码（例如新的⼦类或实现类）来实现，⽽不是去修改已有的、⼯作良好的代码。这是⾯向对象设计中最重要的⼀条原则，是实现系统可复⽤性和可维护性的基⽯。

# 3. ⾥⽒替换原则 (Liskov Substitution Principle, LSP)

核⼼思想：所有引⽤基类的地⽅，都必须能够透明地使⽤其⼦类的对象。

通俗来说：⼦类应当可以完全替换⽗类，并且程序的⾏为不会产⽣任何错误或异常。这意味着⼦类在继承⽗类时，不应该改变⽗类预期的⾏为。它是实现开闭原则的重要⽅式之。

# 4. 接⼝隔离原则 (Interface Segregation Principle, ISP)

核⼼思想：客户端不应该被强迫依赖它不需要的接⼝。⼀个类对另⼀个类的依赖应该建⽴在最⼩的接⼝之上。

通俗来说：接⼝的设计要“专”，不要设计⼤⽽全的“胖接⼝”，⽽应该根据功能设计多个细粒度的接⼝。这样可以避免⼀个类为了实现接⼝中的某⼀个功能，⽽不得不实现所有其他它并不需要的功能。

# 5. 依赖倒置原则 (Dependency Inversion Principle, DIP)

核⼼思想：⾼层模块不应该依赖于低层模块，两者都应该依赖于抽象；抽象不应该依赖于细节，细节应该依赖于抽象。

通俗来说：我们要⾯向接⼝编程，⽽不是⾯向实现编程。代码的依赖关系应该通过抽象（如接⼝或抽象类）来建⽴，⽽不是具体的实现类。这样可以⼤⼤降低模块间的耦合度，⽅便系统的升级和扩展。

# 其他重要原则：

1. 合成/聚合复⽤原则 (Composite/Aggregate Reuse Principle, CARP)

核⼼思想：尽量使⽤对象组合（合成/聚合）的⽅式，⽽不是继承来达到代码复⽤的⽬的。

通俗来说：“has-a” 关系通常⽐ “is-a” 关系更灵活。继承是⼀种强耦合关系（⽩盒复⽤），⽽组合是⼀种弱耦合关系（⿊盒复⽤），后者更容易维护和扩展。

2. 迪⽶特法则 (Law of Demeter, LoD) / 最少知识原则 (Least Knowledge Principle, LKP)

核⼼思想：⼀个软件实体应当尽可能少地与其他实体发⽣相互作⽤。

通俗来说：⼀个对象应该对其他对象有尽可能少的了解，即“只和你的直接朋友交谈”。这有助于降低类之间的耦合度，让每个模块更加独⽴。

这些原则并⾮孤⽴存在，⽽是相辅相成，共同指导我们写出“⾼内聚、低耦合”的优质代码。

# 什么是设计模式？

设计模式（Design Pattern）可以理解为⼀套在特定场景下，针对软件设计中常⻅问题的、可复⽤的解决⽅案。

它不是⼀个可以直接转换成代码的最终设计，⽽更像⼀个蓝图或模板，描述了如何组织类和对象来解决某个特定的设计难题。

我们可以从以下⼏个⻆度来理解它：

1. 它是经验的总结：设计模式是⽆数软件⼯程师在⻓期实践中，经过反复试验和验证，总结出的⼀套⾏之有效的“最佳实践”。

2. 它是沟通的语⾔：设计模式为开发者提供了⼀套通⽤的词汇。当你说“这⾥我⽤了⼀个⼯⼚模式”，团队⾥的其他成员能⽴刻理解你的设计意图，这极⼤地提⾼了沟通效率。

3. 它解决了什么问题：它的核⼼⽬标是提升代码的可重⽤性、可读性、健壮性和可维护性。通过遵循设计模式，我们可以写出结构更清晰、耦合度更低、更容易扩展和修改的代码，避免重复“造轮⼦”和踩⼀些常⻅的“坑”。

总⽽⾔之，设计模式是软件⼯程化的重要基⽯，它帮助我们将代码设计从“个⼈技艺”提升到“⼯程规范”的层⾯。

# 设计模式的分类了解吗?

根据设计模式的⽬的和关注点的不同，它们通常被分为三⼤类，这是由《设计模式：可复⽤⾯向对象软件的基础》这本书（常被称为 "GoF 四⼈帮" 的著作）提出的经典分类：

# 1. 创建型模式 (Creational Patterns)

关注点：对象的创建过程。这类模式将对象的创建和使⽤解耦，使得程序在创建对象时更具灵活性。它们隐藏了对象创建的复杂逻辑，使得我们⽆需直接使⽤ new 关键字。

核⼼思想：提供⼀种机制，使得客户端代码不必关⼼它所需要的具体是哪个类的实例，也不必关⼼这些实例是如何被创建和组织的。

#  常⻅模式：

单例模式 (Singleton)

⼯⼚⽅法模式 (Factory Method)

抽象⼯⼚模式 (Abstract Factory)

建造者模式 (Builder)

原型模式 (Prototype)

# 2. 结构型模式 (Structural Patterns)

关注点：类和对象的组合。这类模式研究如何将类和对象组合在⼀起，形成更⼤、更复杂的结构，同时保持结构的灵活性和效率。

核⼼思想：通过继承、组合等⽅式，在不改变原有类的情况下，为其增加新的功能或适配不同的接⼝。

# 常⻅模式：

适配器模式 (Adapter)

桥接模式 (Bridge)

组合模式 (Composite)

装饰器模式 (Decorator)

外观模式 (Facade)

享元模式 (Flyweight)

代理模式 (Proxy)

# 3. ⾏为型模式 (Behavioral Patterns)

关注点：对象之间的通信和职责分配。这类模式专⻔处理对象之间的交互和协作，以及算法和责任的分配。

核⼼思想：通过定义对象间的通信模式来降低它们之间的耦合度，使得系统中的对象可以独⽴地变化。

#  常⻅模式：

责任链模式 (Chain of Responsibility)

命令模式 (Command)

迭代器模式 (Iterator)

中介者模式 (Mediator)

备忘录模式 (Memento)

观察者模式 (Observer)

状态模式 (State)

模板⽅法模式 (Template Method)

访问者模式 (Visitor)

解释器模式 (Interpreter)

# ⼯⼚模式

# 请谈⼀谈你对⼯⼚模式的理解

⼯⼚模式是创建型设计模式中最常⽤的⼀族，它主要解决的是对象的创建问题，将对象的创建和使⽤过程进⾏解耦。根据其复杂度和应⽤场景的不同，通常分为三种：

1. 简单⼯⼚模式

2. ⼯⼚⽅法模式

3. 抽象⼯⼚模式

# 说⼀说简单⼯⼚模式

简单⼯⼚模式指由⼀个单⼀的⼯⼚对象来创建实例，客户端不需要关注创建逻辑，只需提供传⼊⼯⼚的参数。

UML 类图如下：

![image](https://cdn-mineru.openxlab.org.cn/result/2026-03-20/9b458c1b-11e2-4de5-b51e-4087461b6679/c9174b2259fdaade4d19282e751be7fadbf80c50e66fb32d8bda7f38fbfe978f.jpg)


⻆⾊组成：

⼯⼚ (Factory)：负责实现创建所有实例的内部逻辑。

。 抽象产品 (AbstractProduct)：所有被创建对象的⽗类或接⼝。

具体产品 (ConcreteProduct)：⼯⼚类创建的⽬标实例。

解决的问题：将客户端代码与具体产品的实现类解耦。客户端只需“消费”产品，⽽⽆需关⼼产品是如何被创建的。

. 优缺点：

优点：结构简单，将创建逻辑集中管理，实现了职责分离。

缺点：违反了开闭原则。当需要增加新产品时，必须修改⼯⼚类内部的判断逻辑（例如 if-else 或 switch 语句），这使得⼯⼚类越来越臃肿，难以维护。

实例：

Java 的 Calendar.getInstance() ⽅法，内部会根据时区、地区等参数创建具体的⽇历对象。

Spring 的 BeanFactory 在概念上就是⼀个强⼤的简单⼯⼚，你通过⼀个 bean 的名字（ID）就能获取到对应的实例，⽽⽆需关⼼这个 bean 是如何被创建和配置的。

# 静态⼯⼚和简单⼯⼚的区别

静态⼯⼚（Static Factory） 是⼀种基于类的静态⽅法创建对象的⽅式，属于《Effective Java》中推荐的创建对象的技巧（严格来说不算标准设计模式，但常被归类为创建型模式）

上⾯的简单⼯⼚模式代码⽤静态⼯⼚改写如下：

```txt

    void operate(); // 操作设备
}
// 具体产品：手机（包含静态工厂方法）

    // 私有构造函数，强制通过静态工厂创建
    private Phone() {}
}
// 静态工厂方法：创建手机实例
public static Phone createPhone() {
    return new Phone();
}
}
@override
public void operate() {
    System.out.println("手机操作：开机 -> 显示主界面 -> 关机\n");
}
```

# ⼯⼚⽅法模式了解吗？

为了解决简单⼯⼚模式违反开闭原则的问题，⼯⼚⽅法模式将创建具体对象的任务下放给⼦类。它定义了⼀个⽤于创建对象的抽象⽅法，但由⼦类来决定要实例化哪⼀个类。

UML 类图如下：

![image](https://cdn-mineru.openxlab.org.cn/result/2026-03-20/9b458c1b-11e2-4de5-b51e-4087461b6679/9a207a6ed603ad82c20507fd16cce96ef46ef8ace477884c2c856ab30287480b.jpg)


⻆⾊组成：

抽象⼯⼚ (AbstractFactory)：声明了创建产品的抽象⽅法。

具体⼯⼚ (ConcreteFactory)：实现抽象⽅法，负责创建具体的产品。

C 抽象产品 (AbstractProduct)：与简单⼯⼚模式相同。

。 具体产品 (ConcreteProduct)：与简单⼯⼚模式相同。

解决的问题：在不修改现有⼯⼚类的情况下，轻松扩展新的产品。如果想增加⼀个新产品，只需增加⼀个新的具体产品类和⼀个对应的具体⼯⼚类即可，完全符合开闭原则。

. 优缺点：

优点：完美遵循开闭原则，扩展性好。创建逻辑被分散到各个具体⼯⼚中，符合单⼀职责原则。

缺点：每增加⼀个产品，就需要增加⼀个具体⼯⼚类，这会导致系统中的类数量成倍增加，增加了系统的复杂性。

实例：Java 集合框架中 Collection 接⼝的 iterator() ⽅法。 ArrayList 和 LinkedList 都实现了这个接⼝，但它们各⾃的 iterator() ⽅法返回的是不同的迭代器实现类 ( ArrayListIterator 和LinkedListIterator )，这就是典型的⼯⼚⽅法模式。

# 抽象⼯⼚模式了解吗？

当需要创建的不是单⼀产品，⽽是⼀个产品族（⼀系列相互关联或相互依赖的对象）时，就该使⽤抽象⼯⼚模式。它提供⼀个接⼝，⽤于创建⼀系列相关或相互依赖的对象，⽽⽆需指定它们具体的类。可以理解为它是“⼯⼚的⼯⼚”。

# "产品族"和 "产品等级结构"概念介绍：

产品等级结构：同⼀类产品的不同实现（如按钮可以有 Windows 按钮、Mac 按钮，这构成⼀个产品等级结构）

产品族：同⼀品牌或⻛格下的不同产品（如 Windows ⻛格下的按钮、⽂本框、复选框，构成⼀个产品族）

![image](https://cdn-mineru.openxlab.org.cn/result/2026-03-20/9b458c1b-11e2-4de5-b51e-4087461b6679/dd21b1b073834dc60d8f577395b4227f309c32419887f3dd0e04f191b989d56e.jpg)


# ⻆⾊组成：

抽象⼯⼚ (AbstractFactory)：定义了创建⼀系列不同产品（⼀个产品族）的接⼝。

具体⼯⼚ (ConcreteFactory)：实现接⼝，创建特定主题或⻛格的产品族。

抽象产品 (AbstractProduct)：为产品族中的每⼀种产品定义接⼝。

具体产品 (ConcreteProduct)：实现抽象产品接⼝，是具体⼯⼚创建的⽬标。

解决的问题：解决创建⼀整个产品家族的问题，保证客户端在切换不同产品族时，能获得⼀整套相互兼容和匹配的对象。

# . 优缺点：

优点：⾮常适合⽤于创建⼀系列相互匹配的产品。切换整个产品族⾮常⽅便，只需更换具体的⼯⼚即可。

缺点：扩展新的产品等级结构困难。例如，如果产品族需要增加⼀个“⿏标”，那么AbstractFactory 接⼝就需要增加⼀个 createMouse() ⽅法，所有已经实现的具体⼯⼚类也都要跟着修改，这违反了开闭原则。

# 单例模式

# 什么是单例模式？有什么优点？

单例模式属于创建型模式，⼀个单例类在任何情况下都只存在⼀个实例，构造⽅法必须是私有的、由⾃⼰创建⼀个静态变量存储实例，对外提供⼀个静态公有⽅法获取实例。

# 优缺点：

优点：

节省资源：由于内存中只有⼀个实例，减少了频繁创建和销毁对象带来的性能开销。

全局控制：⽅便对唯⼀实例进⾏统⼀管理，避免对共享资源的多重占⽤。

. 缺点：

。 扩展性差：因为没有抽象层，难以扩展。

职责过重：⼀个类既要负责⾃身的业务逻辑，⼜要负责保证单例，有点违背单⼀职责原则。

。 测试困难：全局状态在单元测试中可能引⼊依赖和副作⽤。

# 单例模式的常⻅写法有哪些？

单例模式的实现⽅式有很多种，它们主要在线程安全和懒加载（Lazy Loading）这两个维度上有所不同。

# 4. 懒汉式 - 双重检查锁

双重检查锁 (Double-Checked Locking, DCL) 是对同步⽅法懒汉式的优化，它通过两次 null 检查和synchronized 代码块，在保证线程安全的同时，⼤⼤提升了性能。

```java
public class Singleton {
    private Singleton() {}
    // 使用 volatile 保证可见性和禁止指令重排
    private static volatile Singleton instance;
    public static Singleton getInstance() {
        // 第一次检查：避免不必要的同步
        if (instance == null) {
            // 第二次检查：在同步块内部确保只有一个线程创建实例
            synchronized (Singleton.class) {
                if (instance == null) {
                    // new 操作不是原子性的，需要 volatile
                    instance = new Singleton();
                }
            }
        }
    }
    return instance;
}
```

# 双重检查的必要性：

外层检查：当实例已创建时，所有线程可直接返回结果，避免不必要的同步开销，这是性能优化的关键。

内层检查：解决并发竞争问题 —— 当多个线程同时通过外层检查时，同步块保证只有⼀个线程进⼊创建逻辑，后续线程会被内层检查拦截，确保仅创建⼀个实例。

# 5. 静态内部类 - 推荐

虽然 DCL 已经很优秀了，但在现代 Java 开发中，我们有更优雅、更推荐的实现⽅式。

这是我个⼈⽐较推荐的写法，它巧妙地利⽤了 JVM 的类加载机制来实现懒加载和线程安全。

```java
public class Singleton {
    private Singleton() {}
    private static class SingletonHolder {
        private static final SingletonINSTANCE = new Singleton();
    }
    public static Singleton.getInstance() {
        return SingletonHolderINSTANCE;
    }
}
```

静态内部类单例是如何实现懒加载的呢？⾸先，我们先了解下类的加载时机。

对于初始化阶段，虚拟机严格规范了有且仅有 6 种情况下，必须对类进⾏初始化(只有主动去使⽤类才会初始化类)：

1. 遇到 new 、 getstatic 、 putstatic 或 invokestatic 这 4 条字节码指令时：

new : 创建⼀个类的实例对象。

。 getstatic putstatic : 读取或设置⼀个类型的静态字段（被 final 修饰、已在编译期把结果放⼊常量池的静态字段除外）。

。 invokestatic : 调⽤类的静态⽅法。

2. 使⽤ java.lang.reflect 包的⽅法对类进⾏反射调⽤时如 Class.forName("...") , newInstance() 等等。如果类没初始化，需要触发其初始化。

3. 初始化⼀个类，如果其⽗类还未初始化，则先触发该⽗类的初始化。

4. 当虚拟机启动时，⽤户需要定义⼀个要执⾏的主类 (包含 main ⽅法的那个类)，虚拟机会先初始化这个类。

5. MethodHandle 和 VarHandle 可以看作是轻量级的反射调⽤机制，⽽要想使⽤这 2 个调⽤，就必须先使⽤ findStaticVarHandle 来初始化要调⽤的类。

6. 「补充，来⾃issue745」 当⼀个接⼝中定义了 JDK8 新加⼊的默认⽅法（被 default 关键字修饰的接⼝⽅法）时，如果有这个接⼝的实现类发⽣了初始化，那该接⼝要在其之前被初始化。

关于类加载的详细介绍可以阅读笔者写的这篇⽂章：类加载过程详解。

这 6 种情况被称为是类的主动引⽤，注意，这⾥《虚拟机规范》中使⽤的限定词是 "有且仅有"，那么，除此之外的所有引⽤类都不会对类进⾏初始化，称为被动引⽤。静态内部类就属于被动引⽤的情况。

当 getInstance() ⽅法被调⽤时， InnerClass 才在 Singleton 的运⾏时常量池⾥，把符号引⽤替换为直接引⽤，这时静态对象 INSTANCE 也真正被创建，然后再被 getInstance() ⽅法返回出去，这点同饿汉模式。

那么 INSTANCE 在创建过程中⼜是如何保证线程安全的呢？在《深⼊理解 JAVA 虚拟机》中，有这么⼀句话:

虚拟机会保证⼀个类的 <clinit>() ⽅法在多线程环境中被正确地加锁、同步，如果多个线程同时去初始化⼀个类，那么只会有⼀个线程去执⾏这个类的 <clinit>() ⽅法，其他线程都需要阻塞等待，直到活动线程执⾏ <clinit>() ⽅法完毕。如果在⼀个类的 <clinit>() ⽅法中有耗时很⻓的操作，就可能造成多个进程阻塞(需要注意的是，其他线程虽然会被阻塞，但如果执⾏<clinit>() ⽅法后，其他线程唤醒之后不会再次进⼊ <clinit>() ⽅法。同⼀个加载器下，⼀个类型只会初始化⼀次。)，在实际应⽤中，这种阻塞往往是很隐蔽的。

从上⾯的分析可以看出 INSTANCE 在创建过程中是线程安全的，所以说静态内部类形式的单例可保证线程安全，也能保证单例的唯⼀性，同时也延迟了单例的实例化。

# 适配器模式

# 适配器模式了解吗？

适配器模式（Adapter Pattern）是⼀种结构型设计模式，它的核⼼作⽤是将⼀个类的接⼝转换成客户端所期望的另⼀个接⼝，从⽽使原本因接⼝不兼容⽽不能在⼀起⼯作的类可以协同⼯作。

你可以把它想象成⼀个我们⽇常⽣活中常⽤的电源适配器或转换头。⽐如，你的笔记本电脑电源是两脚插头（被适配者 Adaptee），但墙上的插座是三孔的（⽬标 Target）。这时，你需要⼀个转换头（适配器 Adapter），它⼀端能插进三孔插座，另⼀端能接收你的两脚插头，这样问题就解决了。

适配器模式主要包含三个核⼼⻆⾊：

Target (⽬标接⼝)：客户端（Client）期望和它直接交互的接⼝。在上⾯的例⼦⾥，就是那个三孔插座。

Adaptee (被适配者)：已存在的、但接⼝与 Target 不兼容的类。也就是那个两脚插头。

Adapter (适配器)：模式的核⼼。它实现了 Target 接⼝，同时内部包装了⼀个 Adaptee 类的实例，负责将对 Target 接⼝的调⽤转换为对 Adaptee 接⼝的调⽤。它就是那个转换头。

# 优缺点：

. 优点：

增强了类的复⽤性：可以复⽤已存在的、功能强⼤的 Adaptee 类，⽽⽆需修改其源码。

提⾼了灵活性和扩展性：可以⾮常⽅便地替换或增加新的适配器，来适配不同的Adaptee ，符合开闭原则。

解耦：将客户端（Client）与具体的实现类（Adaptee）解耦，客户端只需要和⽬标接⼝（Target）打交道。

. 缺点：

增加了系统的复杂性：每适配⼀个类都需要增加⼀个适配器类，如果过度使⽤，会导致系统中的类数量增多，代码可读性有所下降。

（针对类适配器）限制较多：由于语⾔的单继承限制，类适配器⼀次最多只能适配⼀个Adaptee 类，并且要求 Target 必须是接⼝或抽象类。

特点：由于 Java 是单继承，这意味着 Adapter 只能适配⼀个 Adaptee 类。它的耦合度相对较⾼。

# 代理模式

# 什么是代理模式？

代理模式（Proxy Pattern）是⼀种结构型设计模式。它的核⼼思想是为其他对象提供⼀种代理，以控制对这个对象的访问。

你可以把它想象成⽣活中的“明星经纪⼈”。粉丝（客户端 Client）不能直接联系到明星（真实对象RealSubject），⽽是需要通过经纪⼈（代理 Proxy）。这个经纪⼈可以帮明星处理很多事情，⽐如过滤掉不重要的请求、安排⽇程、谈合同等等，⽽明星本⼈则可以专注于⾃⼰的核⼼⼯作——表演。

在这个过程中，经纪⼈和明星都实现了相同的“能⼒接⼝”（⽐如“接受采访”、“商业演出”），所以对外界来说，与经纪⼈沟通和与明星直接沟通，在接⼝上是⼀致的。

![image](https://cdn-mineru.openxlab.org.cn/result/2026-03-20/9b458c1b-11e2-4de5-b51e-4087461b6679/46d879386f9d3d1efd7a918ece55d48dece8cf42d876b42ba85a301e21c83a3a.jpg)


代理模式主要包含三个核⼼⻆⾊：

Subject (抽象主题)：定义了真实对象和代理对象共同的接⼝。

RealSubject (真实主题)：被代理的实际对象，它执⾏业务的核⼼逻辑。

Proxy (代理)：持有对真实对象的引⽤，并实现了抽象主题接⼝。它可以在调⽤真实对象前后执⾏额外的操作。这就是“经纪⼈”。

# 为什么要使⽤代理模式？

代理模式的价值在于，它可以在不改变真实对象代码的前提下，为其增加额外的功能。这些功能通常是通⽤的、与核⼼业务⽆关的，例如：

1. 远程代理 (Remote Proxy)：为⼀个位于不同地址空间的对象提供本地的代表。它可以隐藏⽹络通信的细节，使得客户端调⽤远程对象就像调⽤本地对象⼀样。例如，Dubbo、gRPC 等RPC 框架的客户端存根（Stub）就是典型的远程代理。

2. 虚拟代理 (Virtual Proxy)：根据需要创建开销很⼤的对象。如果真实对象创建和初始化⾮常耗时，虚拟代理可以延迟它的创建，直到客户端真正需要它时才实例化。⽐如，加载⼀个⾼清⼤图，可以先显示⼀个占位符（代理），在后台真正加载图⽚（真实对象）。

3. 保护代理/安全代理 (Protection Proxy)：控制对真实对象的访问权限。代理可以根据调⽤者的权限，决定是否将请求转发给真实对象。例如，Spring Security 中实现的⽅法级别的权限控制。

4. 智能引⽤ (Smart Reference)：在访问对象时执⾏⼀些附加操作，如缓存、⽇志记录、事务管理等。这在 Spring AOP 中体现得淋漓尽致。

# 观察者模式

# 说⼀说观察者模式

观察者模式是⼀种⾮常经典和实⽤的⾏为型设计模式。它的核⼼思想在于定义了⼀种⼀对多的依赖关系：当⼀个对象（我们称之为“被观察者”或“主题”）的状态发⽣改变时，所有依赖于它的对象（即“观察者”）都会⾃动收到通知并进⾏相应的更新。

这种模式的本质是解耦，它将被观察者和观察者分离开来，使得它们可以独⽴地变化和复⽤，⽽不需要知道对⽅的具体实现细节。

UML 类图：

![image](https://cdn-mineru.openxlab.org.cn/result/2026-03-20/9b458c1b-11e2-4de5-b51e-4087461b6679/57f4e8cc782e1a96c95f6a8bd192b0b432a4f07c54ccaf12ecedda486f522d3a.jpg)


观察者模式通常包含四个核⼼⻆⾊：

主题/被观察者 (Subject): 这是⼀个接⼝或抽象类，它负责维护⼀个观察者列表，并提供添加、删除观察者的接⼝。最关键的是，它还定义了通知所有观察者的 notify() ⽅法。

具体主题/具体被观察者 (ConcreteSubject): 它是Subject的具体实现。它包含了业务逻辑，并在⾃身状态发⽣变化时，调⽤继承⾃Subject的 notify() ⽅法，通知所有已注册的观察者。

观察者 (Observer): 同样是⼀个接⼝或抽象类，它定义了⼀个 update() ⽅法。当观察者接收到来⾃主题的通知时，这个⽅法就会被调⽤。

具体观察者 (ConcreteObserver): 它是Observer的具体实现。在 update() ⽅法中，它会根据收到的通知，完成具体的业务逻辑，⽐如更新⾃身状态、执⾏某个操作等。

# 优缺点：

. 优点：

⾼度解耦： 这是它最⼤的优点。主题和观察者之间是松耦合的，主题只知道它有⼀系列观察者，但不需要知道它们是谁、做什么。这为系统带来了极⼤的灵活性。

符合开闭原则： 系统是“对扩展开放，对修改关闭”的。当需要增加新的响应⾏为时，我们只需要创建⼀个新的具体观察者类并注册它即可，完全不需要修改主题的代码。

⽀持⼴播通信： 主题可以向所有注册的观察者⼴播通知，这在很多场景下⾮常⾼效。

. 缺点：

1. 潜在的性能问题： 如果观察者数量⾮常多，或者某个观察者的 update ⽅法逻辑复杂、耗时较⻓，那么通知过程可能会导致主线程阻塞，影响性能。在某些场景下，可以考虑使⽤异步通知来优化。

2. 可能导致意外的级联或循环： 如果观察者之间存在复杂的依赖关系，⼀个观察者的更新可能会触发另⼀个观察者（它同时也是另⼀个主题）的更新，形成复杂的调⽤链。如果设计不当，甚⾄可能导致循环调⽤，造成系统崩溃。

3. 调试困难： 由于其松耦合的特性，当程序出现问题时，有时很难追踪到⼀个事件发⽣后，具体是哪个观察者的⾏为导致了问题。

补充⼀下：观察者模式在不同框架中可能有不同的命名⻛格，例如：

Java 中的 Observable （主题）和 Observer （观察者）；

Spring 中的 ApplicationEvent （事件，对应主题状态）和 ApplicationListener （监听器，对应观察者）。

# 你的项⽬是怎么⽤的观察者模式？

在我的项⽬中，观察者模式得到了⼴泛应⽤，⼀个⾮常典型的例⼦就是⽀付成功后的业务处理。

场景是这样的： 在⼀个电商系统中，当⽤户⽀付成功后，我们需要触发⼀系列独⽴的后续操作，⽐如：

更新订单状态为“已⽀付”。

给⽤户的账户增加积分。

通知仓储系统准备发货。

给⽤户发送⼀封确认邮件。

如果不⽤设计模式， 我们可能会在⽀付成功的⽅法⾥，把这四个操作串⾏地写下来。这样做的问题显⽽易⻅：⽀付核⼼逻辑与各种业务逻辑紧紧地耦合在⼀起，每次新增⼀个类似“赠送优惠券”的需求，都必须去修改这个已经很庞⼤和脆弱的核⼼⽅法，这严重违反了开闭原则。

# 我们的解决⽅案是：

我们将“⽀付成功”这个事件抽象为具体主题 (ConcreteSubject)。⽽上述的“更新订单”、“增加积分”、“通知仓库”、“发送邮件”等操作，则分别被设计成独⽴的具体观察者 ( ConcreteObserver )。

# ⼯作流程是：

在系统初始化时，这些观察者会把⾃⼰注册到“⽀付成功”这个主题上。当⽀付⽹关回调我们的接⼝，确认⽀付成功后，主题的 setState() ⽅法被调⽤，然后它会⽴即执⾏ notify() ，遍历内部的观察者列表，并依次调⽤它们的 update() ⽅法。

# 这样做的价值在于：

⾼度解耦： ⽀付核⼼流程完全不关⼼后续有哪些业务，它只负责发布“我成功了”这个消息。

极强的扩展性： 未来如果需要增加“⽀付成功后赠送⼀张抽奖券”的新功能，我们只需要新增个 LotteryObserver 类，并在系统启动时完成注册即可，原有代码⼀⾏都不⽤动。

# 装饰器模式

# 什么是装饰器模式？

装饰器模式是⼀种结构型模式，它的核⼼思想是在不改变原有对象结构和代码的前提下，动态地为该对象添加额外的功能。

它就像给⼀个物体“穿⾐服”，你可以⼀层⼀层地套上不同的⾐服（装饰器），每⼀件⾐服都增加⼀种新的特性（功能），但物体的本质（核⼼功能）并没有改变。这个过程是通过创建⼀个包裹原始对象的“装饰器”对象来实现的，⽽不是通过继承。

装饰器模式是开闭原则的典范应⽤，它让我们在不修改既有代码的情况下，为对象赋予了⽆限的扩展可能。

装饰器模式通常包含四个核⼼⻆⾊：

Component (抽象构件): 这是⼀个接⼝或抽象类，它定义了原始对象和装饰器对象所共有的接⼝，确保它们可以被⼀致地对待。

ConcreteComponent (具体构件): 这就是我们的“裸对象”，即被装饰的原始对象，它实现了Component 接⼝。

Decorator (抽象装饰器): 它也实现了 Component 接⼝，并且内部持有⼀个 Component 类型的引⽤（即它要装饰的对象）。它的存在是为了将装饰逻辑与基础组件分离开。

ConcreteDecorator (具体装饰器): 这是实际的“⾐服”。它继承⾃ Decorator ，负责向被装饰的对象添加具体的新功能。在实现 Component 接⼝的⽅法时，它除了调⽤原始对象（通过持有的引⽤）的相应⽅法外，还会加上⾃⼰的“装饰”逻辑。

# 讲讲装饰器模式的应⽤场景

场景⼀：运⾏时动态扩展功能。 当我们希望在运⾏时根据不同条件为对象增添不同功能时，装饰器模式是绝佳选择。最经典的例⼦就是Java的I/O流。 FileInputStream 是⼀个具体构件，我们可以⽤ BufferedInputStream 来装饰它，为其增加缓冲功能以提⾼性能；还可以再⽤DataInputStream 来装饰，进⼀步为其增加读取基本数据类型的能⼒。这些装饰器可以任意组

合，⾮常灵活。可以顺带阅读⼀下笔者写的这篇 Java IO 设计模式总结。

场景⼆：当继承⽅案不可⾏或不适⽤时。 如果⼀个类被 final 修饰，我们⽆法通过继承来扩展它。此时，装饰器模式就成了“救星”。此外，如果功能的组合种类⾮常多，使⽤继承会产⽣⼤量的⼦类，导致“类爆炸”问题，⽽装饰器模式则可以⽤少量的类灵活组合出所有需要的功能。

# 责任链模式

# 什么是责任链模式？

责任链模式（Chain of Responsibility Pattern）是⼀种⾏为型设计模式。它将请求的发送者和接收者解耦，通过创建⼀个处理请求的接收者链来处理请求。链上的每个接收者（也称为处理器或节点）都负责对请求进⾏⼀部分的处理或校验，并决定是否将请求传递给链上的下⼀个接收者，或者中断处理流程。

举个例⼦，你提交了⼀个电商订单，这个订单需要经过多个步骤的校验才能完成：库存校验 $\mathrel { - } >$ ⻛控校验 $\mathrel { - } >$ ⽀付信息校验 $\yen 123,456,78$ 订单完成 。每个校验步骤都像链条上的⼀个环节，只有通过当前环节的校验，订单才能进⼊下⼀个环节。任何⼀个环节校验失败，整个订单流程都会终⽌。

责任链模式的结构相对简单：

Handler (处理器): 这是⼀个接⼝或抽象类，定义了处理请求的接⼝（如 handleRequest() ）以及⼀个指向下⼀个处理器的引⽤（ setNext() / getNext() ） 。

ConcreteHandler (具体处理器): 它实现了 Handler 接⼝。在处理⽅法中，它⾸先判断⾃⼰是否能处理当前请求。如果能，就处理它；如果不能，就将请求传递给链上的下⼀个处理器。

# 讲讲责任链模式的应⽤场景

适⽤于多节点的流程处理，每个节点完成各⾃负责的部分，节点之间不知道彼此的存在，⽐如：

订单校验： ⼀个订单可能需要进⾏多种校验，例如商品库存校验、⻛控校验、⽀付信息校验等。可以将这些校验规则组成⼀个责任链，每个校验规则负责⼀种校验，如果校验不通过，则中断流程并返回错误信息；如果校验通过，则将请求传递给下⼀个校验规则。

OA的审批流： 不同的审批级别（例如部⻔经理、总经理等）组成⼀个责任链。每个审批级别都负责审批⼀部分内容，如果审批不通过，则中断流程并返回原因；如果审批通过，则将请求传递给下⼀个审批级别。

Filter（过滤器）： 多个Filter组成⼀个责任链，实现对 HTTP 请求的过滤功能，⽐如鉴权、限流、记录⽇志、验证参数等等。

Interceptor（拦截器）： 类似于Filter，Interceptor也可以拦截请求并在请求处理前后执⾏⼀些操作。

# 策略模式

# 什么是策略模式？

策略模式的核⼼思想是：定义⼀系列算法，将每⼀个算法都封装起来，并使它们可以相互替换。

换句话说，它允许我们在运⾏时，根据不同的情况，动态地改变⼀个对象所使⽤的算法。这个对象（我们称之为“上下⽂/Context”）只依赖于⼀个抽象的策略接⼝，⽽不需要关⼼具体是哪种算法在执⾏。

这就像我们开⻋去⼀个地⽅，可以有好⼏种导航策略：⽐如“时间最短策略”、“距离最短策略”、“避开⾼速策略”。我们的“出⾏”这个⾏为（Context）本身不变，但可以随时切换具体使⽤的导航算法（Strategy）。

# 策略模式有什么好处？

1. 完美遵循开闭原则： 这是策略模式最⼤的价值。当我们需要增加⼀种新的算法（⽐如新的“节假⽇促销策略”）时，我们只需要新增⼀个策略类即可，完全不需要修改使⽤这些策略的上下⽂（Context）代码。这使得系统扩展性极强。

2. 算法的独⽴与解耦： 每个算法都被封装在独⽴的策略类中，使得算法本身可以独⽴于使⽤它的客户端⽽变化。这让代码更清晰，也更容易对每个算法进⾏单独的测试和维护。

3. 避免冗⻓的条件语句： 它将复杂的、与业务逻辑紧密相关的 if-else 或 switch-case 结构，转变为⼀系列清晰、独⽴的策略类，⼤⼤提⾼了代码的可读性和可维护性。

# 什么是状态模式？

状态模式（State Pattern）是⼀种⾏为型设计模式。它的核⼼思想是，允许⼀个对象的⾏为在内部状态改变时随之改变，这个对象看起来就像是修改了它的类。

简单来说，状态模式将与特定状态相关的⾏为局部化，并将不同状态的⾏为分割到不同的状态类中。Context（上下⽂）对象将⾏为委托给当前的状态对象，从⽽消除了原来 Context 中庞⼤的条件分⽀语句（ if-else 或 switch-case ）

# 优缺点：

. 优点：

结构清晰：将不同状态的⾏为封装到各⾃的类中，使得代码结构更清晰，职责更单⼀。

易于扩展：增加新的状态⾮常⽅便，只需增加⼀个新的具体状态类即可，符合“开闭原则”。

封装转换规则：状态之间的转换逻辑可以被封装在状态类内部，减少了上下⽂类的负担。

避免条件滥⽤：有效避免了冗⻓的 if-else 或 switch 语句，提⾼了代码的可维护性。

缺点：

类数量增多：每个状态都需要⼀个对应的类，当状态过多时，会导致系统中类的数量增加。

实现相对复杂：相⽐于简单的条件判断，状态模式的结构和实现逻辑更复杂。

状态转换耦合：如果状态转换逻辑分散在各个具体状态类中，那么在增加新状态时，可能需要修改已有状态类的转换逻辑，这在⼀定程度上违反了“开闭原则”。

# 状态模式与策略模式的区别

状态模式和策略模式在结构上⾮常相似（都有⼀个上下⽂类、⼀个抽象接⼝和多个具体实现类），但它们的意图完全不同：

状态模式：关注的是状态的⾃动转换。状态的变更通常由状态内部逻辑决定，对客户端是透明的。它旨在解决对象在不同状态下⾏为不同的问题，强调的是“下⼀个状态是什么”。

策略模式：关注的是策略的灵活选择。具体使⽤哪种策略由客户端（或上层逻辑）决定并主动设置，策略之间通常没有直接关联。它旨在解决同⼀⾏为有不同实现算法的问题，强调的是“如何做”。

# 状态机了解吗？

状态机（State Machine），全称为有限状态⾃动机（Finite State Automaton），是⼀个⽤来描述系统在特定输⼊信号或事件下如何从⼀个状态转换到另⼀个状态的数学模型。在任何特定时间点，系统都处于某⼀确定的状态，⽽状态机定义了这些状态之间的转换规则。

状态机可以被视为⼀种设计模式，通常称为状态模式（State Pattern） 。

状态机的引⼊可以使业务模型更加清晰，帮助开发⼈员更好地理解和实现业务逻辑。此外，状态机的使⽤有利于代码的维护和扩展。然⽽，状态机的引⼊也可能增加编码复杂性，对开发⼈员的编程技能提出了更⾼的要求。

状态机的四⼤概念：

1. 状态（State）：状态机⾄少包含两个状态。例如，⼀个订单的状态可能包括“待⽀付”、“待发货”、“待收货”、“待评价”等。

2. 事件（Event）：事件是触发状态转换的条件或指令。对于订单系统，事件可以是“⽀付成功”、“发货”、“确认收货”、“完成评价”等。

3. 动作（Action）：在事件发⽣后将执⾏的动作，通常是调⽤某些⽅法。例如，在客户确认收货后，系统需执⾏更新订单状态到“待评价”的操作。

4. 转换（Transition）：转换描述了⼀个状态如何基于某个事件转变到另⼀个状态。例如，当“⽀付成功”事件发⽣时，可能触发“处理⽀付”动作，并将订单状态从“待⽀付”转变为“待发货”。

如果业务流程包含超过三个状态（例如订单状态、商品状态），就可以考虑是否使⽤状态机了。

状态机与流程引擎的主要区别：

1. 状态机关注于对象的状态转换；流程引擎更多地是关注整个业务流程的组织和执⾏。

2. 状态机结构相对简单，特别适合清晰、固定的流程；流程引擎⽀持多层次、多路径的复杂流程，允许动态调整和扩展。

3. 状态机⾃动根据输⼊或事件改变状态，不需要⼈⼯⼲预；流程引擎则常涉及⼈的参与决策和协同⼯作。

在复杂系统中，流程引擎负责⾼层次的流程管理，⽽状态机处理具体对象的状态转换，两者互补，共同提⾼系统的效率和可维护性。

对于 Spring 项⽬来说，Spring StateMachine 就是开箱即⽤的状态机实现，功能⾮常强⼤。不过，Spring StateMachine 上⼿难度⽐较⼤，还会增加项⽬的复杂性（有点笨重）。因此，对于状态机功能性需求要求不⾼的场景（绝⼤部分项⽬要求都不⾼的），不建议使⽤。并且，由于 SpringStateMachine 状态机实例不是⽆状态的（包含了上下⽂），⽆法做到线程安全，所以代码要么需要使⽤锁同步，要么需要⽤ Threadlocal ，⾮常的痛苦和难⽤。 每⼀次状态机在接收请求的时候，都不得不重新 Build ⼀个新的状态机实例，导致性能也会⽐较差，浪费资源。

这⾥推荐另外⼀款阿⾥⼤佬开源的状态机 Cola-StateMachine，这是⼀种轻量级的、⽆状态的、基于注解的状态机实现。由于⽆状态的特点，可以做到线程安全。

Cola-StateMachine 使⽤ Java 实现，最简单，实现成本也最低，但是不⽀持“外部配置”。

# Netty 使⽤了哪些设计模式？

Netty 作为⾼性能的⽹络通信框架，在源码中⼤量运⽤设计模式以保证代码的灵活性、可扩展性和可维护性：

单例模式： DefaultSelectStrategy （选择策略）通过 static final INSTANCE 直接初始化，类加载时创建唯⼀实例，供所有 NioEventLoop 共享，避免了重复创建。

⼯⼚⽅法模式： ReflectiveChannelFactory 作为 Channel 的⼯⼚类，通过反射根据传⼊的 Class参数动态创建不同类型的 Channel （如服务端 NioServerSocketChannel 、客户端NioSocketChannel ），避免为每种 Channel 单独创建⼯⼚类。

责任链模式：Netty 将⽹络事件（如连接建⽴、数据读写）的处理流程，抽象成了⼀条处理链（ Pipeline ）。这条链由⼀个个的处理器（ ChannelHandler ）组成，每个处理器负责⼀项特定的任务，⽐如解码、编码、业务逻辑处理等。当⼀个事件进来时，它会沿着这条链，从头到尾依次被处理器处理。

观察者模式：Netty的所有I/O操作都是异步的。那么，我们如何知道⼀个异步操作（⽐如发送数据）何时完成、是成功还是失败呢？Netty 通过 ChannelFuture 解决了这个问题。ChannelFuture 作为被观察者，通过 addListener() 注册观察者（ GenericFutureListener ），当异步操作（如 writeAndFlush ）完成时，⾃动触发所有监听器的 operationComplete() ⽅法，回调处理结果。

建造者模式： ServerBootstrap 与 Bootstrap 作为客户端和服务端的引导器，通过链式⽅法配置核⼼参数（如 group() 设置线程池、 channel() 指定 Channel 类型、 childHandler() 注册处理器），最终通过 bind() 或 connect() 启动，清晰且灵活。

策略模式： EventExecutorChooser 根据 EventLoopGroup 中线程池的⼤⼩（是否为 2 的幂），动态选择 PowerOfTwoEventExecutorChooser （⾼效取模）或 GenericEventExecutorChooser （普通取模），优化 EventLoop 的选择性能。

装饰者模式： WrappedByteBuf 作为所有 ByteBuf 装饰器的基类，包装原始 ByteBuf 实例。⼦类如 UnreleasableByteBuf 增强 release() ⽅法（返回 false 禁⽌释放），SimpleLeakAwareByteBuf 增加内存泄漏检测，在不修改原 ByteBuf 的前提下扩展功能。

# Dubbo 中⽤到了哪些设计模式？

Apache Dubbo 作为成熟的微服务框架，其源码通过⼤量设计模式解决了分布式场景下的灵活性、可扩展性和耦合性问题：

单例模式：作为 Dubbo SPI 机制的核⼼， ExtensionLoader 通过 ConcurrentMap 缓存每种类型的加载器实例，确保对同⼀扩展点类型仅存在⼀个加载器，避免重复创建导致的资源浪费和状态不⼀致。

⼯⼚模式： ExtensionLoader#createExtension 根据名字实例化具体扩展类属于简单⼯⼚，ProxyFactory#getProxy / getInvoker ，由 JavassistProxyFactory 、 StubProxyFactoryWrapper 等⼦类决定创建逻辑属于⼯⼚⽅法。

代理模式：当我们在消费端引⽤⼀个远程服务时，我们得到的其实不是⼀个真实的远程对象，⽽是⼀个本地代理对象。 ProxyFactory （通常是 JavassistProxyFactory ）负责创建这个代理。我们对这个代理对象的⽅法调⽤，会被代理逻辑拦截，然后封装成 RPC 请求发送出去。这使得远程调⽤对⽤户来说就像本地调⽤⼀样透明，是实现RPC的基础。

模板⽅法模式：为了规范各种协议（ Protocol ）和注册中⼼（ Registry ）的实现，Dubbo提供了⼤量的抽象基类。这些基类使⽤了模板⽅法模式，定义了核⼼流程的⻣架（⽐如 export 、refer register ），并把其中易变的部分（如 doExport 、 doRefer ）声明为抽象⽅法，交由⼦类去实现。

装饰器模式： MockClusterInvoker （Mock 装饰器）包装原始 Invoker ，在服务调⽤失败时⾃动触发 mock 逻辑（如返回预设结果），增强了容错能⼒。装饰器与原始 Invoker 实现同⼀接⼝，客户端⽆需感知差异。

策略模式： LoadBalance 接⼝定义服务选择算法，具体实现（ RandomLoadBalance 随机、RoundRobinLoadBalance 轮询等）封装不同负载均衡逻辑。框架根据配置（如loadbalance=random ）动态选择策略，⽀持灵活切换。

责任链模式： ProtocolFilterWrapper 通过 buildInvokerChain() 构建过滤器链，将多个 Filter（如监控、⽇志、限流、认证等）按顺序封装为 Invoker 链。请求经过每个过滤器时，可修改参数或中断传递，最终到达⽬标服务。
